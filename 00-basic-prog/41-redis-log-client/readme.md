*how to use it*

1. compile simple redis client example
2. install redis locally like: `sudo apt-get install redis`
3. start redis service like: `sudo systemctl start redis`
4. run ./00-basic-prog/41-redis-log-client/Debug/41-redis-log-client
5. you should see all connected and written to redis
6. check redis with: `redis-cli` && `XRANGE leo_phone_11_03_2026 - +`
7. you should see all logs in output

КРИТИЧЕСКИЕ — сломается даже при умеренной нагрузке
1. Исчерпание пула потоков httplib (самое опасное)
cpp-httplib использует модель "один поток на соединение" с ограниченным пулом (по умолчанию ~8 потоков). Каждое SSE-соединение держит поток навечно:

devices_stream_provider — поток висит на wait_for_devices_change (до 15 сек)
log_names_stream_provider — поток висит на wait_for_streams_change (до 15 сек)
logs_stream_provider — поток висит на xread(..., 15000) (до 15 сек)
Один браузер создает 3 SSE-соединения. Открыли 3 вкладки — это уже 9 потоков. Пул исчерпан. После этого сервер не может обработать ни одного запроса — ни статику, ни download, ни новые SSE. Он просто зависает.

2. XRANGE "-" "+" загружает ВСЕ логи в память
redis_client_->xrange(stream_key_, "-", "+", std::back_inserter(stream_data));
Если в стриме 500K записей — все они загрузятся в std::vector, потом сериализуются в JSON, потом отправятся браузеру одним SSE-событием. Это:

сожрет память на сервере
заблокирует Redis-соединение надолго
браузер подвиснет, парся огромный JSON
3. Все логи вставляются в DOM
На клиенте в init все исторические логи добавляются как DOM-элементы. 100K строк лога — это 100K <div>, браузер начнет тормозить или упадет.

4. Единый Redis-клиент на все соединения
Один shared_ptr<redis::Redis> используется всеми SSE-провайдерами и обработчиками одновременно. redis-plus-plus внутри имеет пул соединений, но размер по умолчанию маленький. Под нагрузкой все потоки будут конкурировать за соединения к Redis.

ВЫСОКИЕ — будут проблемы при росте числа устройств/пользователей
5. N+1 запросов к Redis в get_devices_json()
for (const auto& device : devices)
{
    auto platform = redis_client_->hget(
        std::format("device_info:{}", device), "platform");
    ...
}
Для 100 устройств — 1 SMEMBERS + 100 HGET = 101 round-trip к Redis. Решается одним PIPELINE или хранением платформы прямо в all_devices (например через HASH вместо SET).

6. Thundering herd (пробуждение толпой)
cv_streams.notify_all() будит ВСЕ потоки, ожидающие изменений стримов, даже если изменился стрим только одного устройства. 50 клиентов смотрят 50 разных устройств — при изменении одного все 50 проснутся, проверят условие, и 49 снова заснут. Впустую.

7. Нет пагинации ни для чего
SMEMBERS "all_devices" возвращает все устройства
SMEMBERS "log_names:..." возвращает все лог-файлы
Логи грузятся целиком
При тысячах устройств или сотнях лог-файлов на устройство — ответы станут огромными.

8. Redis-стримы растут бесконечно
Ни в клиенте, ни в сервере нет MAXLEN / MINID при XADD. Старые логи никогда не удаляются. Со временем Redis съест всю доступную память.

9. Нет TTL / экспирации
Устройства, лог-файлы, метаданные — живут в Redis вечно. Если устройство перестало работать год назад, оно всё равно отображается в списке.

СРЕДНИЕ — качество / безопасность
10. XSS через innerHTML
div.innerHTML = message;
Если лог-сообщение содержит <script>alert('xss')</script> или любой HTML — он будет исполнен в браузере. Нужно использовать textContent или экранировать HTML.

11. std::localtime в клиенте — не потокобезопасна
Если клиент станет многопоточным, std::localtime возвращает указатель на статический буфер — будет data race. Нужно localtime_r (POSIX) или std::chrono formatting (C++20).

12. SSE не поддерживает Last-Event-Id
При обрыве соединения EventSource автоматически переподключается. Но сервер не обрабатывает заголовок Last-Event-Id, поэтому все логи загружаются заново с нуля.

13. streams_version растет бесконечно
Каждое устройство добавляет запись в unordered_map<string, uint64_t> — она никогда не чистится.
